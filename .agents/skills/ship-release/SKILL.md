# Release Workflow

This project publishes via tag-triggered GitHub Actions. When a tag `v*` is pushed (or a release is created with `gh release create`), `.github/workflows/release.yml` runs Docker build ŌåÆ npm publish in parallel. **Release notes are written by AI (this skill), not auto-generated by GitHub.**

## What this skill does and doesn't do

**Does:**
- Reads commits since last tag, recommends semver bump per conventional commits
- Drafts English release notes, grouped by type, with the user able to edit before publishing
- Bumps `package.json` version, commits as `chore(release): vX.Y.Z`
- Creates the GitHub Release via `gh release create` with the AI-written notes (this also creates and pushes the tag)
- Reports the workflow run URL for the user to monitor

**Doesn't:**
- Publish to npm directly ŌĆö the GitHub Actions workflow does that on tag push
- Build Docker images ŌĆö same, workflow handles it
- Create release notes by hand without AI input ŌĆö the whole point is AI drafts them

## Two modes

- **Preview** ŌĆö just analyze and report; no writes. Use when the user asks "what's in the next release?" or "show me what would go into a release". No version bump, no commit, no tag.
- **Publish** ŌĆö the full flow below. Default when the user asks to release / ship / bump version.

If unclear, ask: "Preview only, or actually publish?"

## Pre-flight checks (always, both modes)

Run in parallel:

```bash
git rev-parse --show-toplevel       # confirm we're in the repo
git branch --show-current           # must be main
git status --porcelain              # must be empty (for publish mode)
git fetch origin main --tags        # sync tags and main
git describe --tags --abbrev=0      # find last tag (may fail if no tags)
gh auth status                      # gh CLI must be authenticated (publish mode)
```

Failure modes:

- **Not on main** ŌĆö in publish mode, refuse. In preview mode, warn but proceed.
- **Dirty working tree** ŌĆö in publish mode, refuse and show `git status` output. Don't auto-stash.
- **Behind origin/main** ŌĆö refuse and tell the user to `git pull --ff-only`.
- **No previous tag** ŌĆö that's the initial release case. Start the version at `v1.0.0` unless `package.json` says otherwise; analyze all commits.
- **`gh` not installed or not authenticated** ŌĆö in publish mode, refuse with install instructions: `brew install gh && gh auth login`.

## Step 1 ŌĆö Analyze commits since last tag

```bash
git log <last-tag>..HEAD --pretty=format:'%H%x09%s%x09%b%x1e' --no-merges
```

Parse each commit as conventional-commits:

- `feat(scope)?: subject` ŌåÆ category **Features**, signal `minor`
- `fix(scope)?: subject` ŌåÆ category **Bug Fixes**, signal `patch`
- `refactor(scope)?: subject` ŌåÆ category **Refactor**, signal `patch`
- `perf(scope)?: subject` ŌåÆ category **Performance**, signal `patch`
- `docs(scope)?: subject` ŌåÆ category **Docs**, signal `patch`
- `chore(scope)?: subject` ŌåÆ category **Chores**, signal `patch`
- `test(scope)?: subject` ŌåÆ category **Tests**, signal `patch`
- `build|ci(scope)?: subject` ŌåÆ category **Build/CI**, signal `patch`
- Non-conforming subject ŌåÆ category **Other**, signal `patch`

**Breaking change detection** (any of the following ŌåÆ signal `major`):
- Subject contains `!` before the colon, e.g. `feat!:` or `feat(api)!:`
- Body or footer contains a line starting with `BREAKING CHANGE:` or `BREAKING-CHANGE:`

## Step 2 ŌĆö Recommend bump

Aggregate signals: `major` > `minor` > `patch`. The highest signal wins.

Compute next version from the last tag (strip leading `v`):
- `major` ŌĆö `(X+1).0.0`
- `minor` ŌĆö `X.(Y+1).0`
- `patch` ŌĆö `X.Y.(Z+1)`

If `package.json` version disagrees with the last tag, mention it; usually the tag is the source of truth.

## Step 3 ŌĆö Collect closed backlog items

Scan `docs/backlog/*.md` for items with `status: done`. These are work items that finished between the last release and now ŌĆö they belong in the release notes as a "Closed items" section so users can see exactly what tracked work shipped.

For each `done` item, capture: `title`, `type`, slug (filename minus `.md`), and `branch` (for the compare link).

If there are zero `done` items, skip the "Closed items" section entirely.

## Step 4 ŌĆö Draft release notes

Generate the notes in this exact shape (English, plain markdown, no emoji unless the user asks):

```markdown
## Highlights

<one or two sentences summarizing the most notable changes ŌĆö what shipped, why it matters. Skip this section if there's only mechanical noise (chore/docs/test only).>

## Closed items
- <title> (<slug>) ŌĆö <type>

## Features
- <subject> (<shortsha>)

## Bug Fixes
- <subject> (<shortsha>)

## Refactor
- <subject> (<shortsha>)

## Performance
- <subject> (<shortsha>)

## Docs
- <subject> (<shortsha>)

## Chores
- <subject> (<shortsha>)

## Breaking Changes
- <description from BREAKING CHANGE footer or commit body>

**Full changelog:** https://github.com/<owner>/<repo>/compare/<last-tag>...<new-tag>
```

Rules:
- Omit any section with zero entries (don't print `## Features` followed by nothing)
- Strip the `type(scope):` prefix from subjects ŌĆö they're already categorized. Capitalize the first letter.
- Use 7-character short SHAs
- Order sections as above (Closed items right after Highlights ŌĆö backlog items are higher-signal than raw commits)
- For the "Highlights" paragraph, look at the actual feature/fix subjects *and* closed backlog item titles, and write a brief human summary ŌĆö don't just list types. If there's nothing exciting, omit Highlights.
- Always include the compare link at the end
- Resolve `<owner>/<repo>` from `git remote get-url origin`

## Step 5 ŌĆö Present and confirm

Show the user, in one message:

1. **Last tag**: `vX.Y.Z` (date)
2. **Recommended bump**: `patch` / `minor` / `major` ŌåÆ **`vA.B.C`**
3. **Rationale**: one sentence on why (e.g. "1 feat commit, no breaking changes ŌåÆ minor bump")
4. **Drafted notes** in a fenced markdown block

Then ask the user:
- Confirm version? (or override: `patch`/`minor`/`major`/explicit `vX.Y.Z`)
- Edit notes? (offer to revise specific sections, or take their full rewrite)

Don't proceed until both are confirmed. If the user edits notes, re-present the final version once for a clean confirmation before publishing.

## Step 6 ŌĆö Publish (only after explicit confirm)

Write the final notes to a temp file:

```bash
NOTES_FILE=$(mktemp -t release-notes-vX.Y.Z.XXXX.md)
# write final notes content to $NOTES_FILE
```

Bump `package.json`:

```bash
# Use the Edit tool ŌĆö don't use sed or jq just to bump a version
```

Then run, sequentially:

```bash
git add package.json
git commit -m "chore(release): vX.Y.Z"
git push origin main
gh release create vX.Y.Z --target main --title "vX.Y.Z" --notes-file "$NOTES_FILE"
```

Why `gh release create` instead of `git tag && git push --tags`:
- It creates the tag remotely and attaches notes in one atomic step
- The `--notes-file` carries the AI-drafted markdown verbatim
- The release workflow triggers on the tag push regardless

`--target main` ensures the release is anchored to the just-pushed commit on main.

## Step 7 ŌĆö Archive closed backlog items

Now that `vX.Y.Z` exists as a tag, move every `docs/backlog/*.md` with `status: done` into `docs/backlog/_done/vX.Y.Z/`. Use `git mv` so the move is tracked cleanly.

```bash
mkdir -p docs/backlog/_done/vX.Y.Z
# for each <slug>.md with status: done:
git mv docs/backlog/<slug>.md docs/backlog/_done/vX.Y.Z/<slug>.md
```

Then commit:

```bash
git add docs/backlog/
git commit -m "chore(backlog): archive items shipped in vX.Y.Z"
git push origin main
```

This commit goes on main *after* the tag, which is fine ŌĆö the tag stays on the version-bump commit, and the archive is housekeeping that lives on the post-tag history.

If there were zero `done` items, skip this step entirely (no empty `_done/vX.Y.Z/` directory, no empty commit).

## Step 8 ŌĆö Report and monitor

After publishing:

1. Print the release URL ŌĆö `gh release view vX.Y.Z --json url --jq .url`
2. Print the workflow run URL ŌĆö `gh run list --workflow=release.yml --limit 1 --json url --jq '.[0].url'`
3. Offer to tail the run: `gh run watch` (don't run automatically ŌĆö it blocks)

Then clean up the temp notes file: `rm -f "$NOTES_FILE"`.

## Initial release special case

If `git describe --tags` returns no tag:

- Default proposed version: read from `package.json`. If that's `0.1.0` or similar pre-1.0, suggest `v0.1.0` or whatever package.json says; let the user override.
- Notes: include all commits, but the "Highlights" section can be a single line like "Initial public release of agents-tracing-dashboard."
- Compare link: link to the full commit history instead of a diff, e.g. `https://github.com/<owner>/<repo>/commits/vX.Y.Z`.

## Edge cases

- **No new commits since last tag** ŌĆö refuse with: "Nothing to release. Last tag vX.Y.Z is already at HEAD." Suggest preview mode if they want to see history.
- **Version in `package.json` ahead of last tag** ŌĆö e.g. tag is `v1.0.7` but `package.json` says `1.1.0`. Surface the mismatch; ask whether to honor `package.json` or compute from the tag.
- **User wants to skip a version** ŌĆö e.g. last tag `v1.0.7`, user asks for `v1.2.0` directly. Allow it; just confirm explicitly.
- **Pre-release tags** (`v1.0.0-rc.1`) ŌĆö out of scope for the default flow. If the user asks for one, do it manually and don't try to auto-bump from a pre-release tag.

## Tone

- Lead with the recommendation, not the analysis ("Recommend `v1.0.8` (patch). Reason: 3 fix commits, no feats.")
- Show the notes in full inside a fenced block ŌĆö don't summarize them, the user is going to read every line
- Keep the confirmation prompt concise: "Confirm `v1.0.8` and these notes? (or: edit notes / pick different bump / cancel)"

## Anti-patterns to avoid

- Don't push the tag first then create the release ŌĆö `gh release create` does both in the right order and the workflow only needs one trigger
- Don't run `npm publish` or `pnpm pack:npm` from this skill ŌĆö the workflow does it. Doubling up causes "version already exists" failures
- Don't `git tag -a vX.Y.Z` locally and `git push --tags` ŌĆö that triggers the workflow without attaching AI-written notes (GH would auto-create a release with empty body)
- Don't bypass the user's confirmation, even on "looks routine" patch bumps. The whole point of this skill is human-in-the-loop notes
- Don't edit `CHANGELOG.md` ŌĆö this project keeps release notes on the GitHub releases page, not in the repo
