# Release Workflow

This project publishes via tag-triggered GitHub Actions. When a tag `v*` is pushed (or a release is created with `gh release create`), `.github/workflows/release.yml` runs Docker build → npm publish in parallel. **Release notes are written by AI (this skill), not auto-generated by GitHub.**

## What this skill does and doesn't do

**Does:**
- Reads commits since last tag, recommends semver bump per conventional commits
- Drafts English release notes, grouped by type, with the user able to edit before publishing
- Bumps `package.json` version, commits as `chore(release): vX.Y.Z`
- Creates the GitHub Release via `gh release create` with the AI-written notes (this also creates and pushes the tag)
- Reports the workflow run URL for the user to monitor

**Doesn't:**
- Publish to npm directly — the GitHub Actions workflow does that on tag push
- Build Docker images — same, workflow handles it
- Create release notes by hand without AI input — the whole point is AI drafts them

## Two modes

- **Preview** — just analyze and report; no writes. Use when the user asks "what's in the next release?" or "show me what would go into a release". No version bump, no commit, no tag.
- **Publish** — the full flow below. Default when the user asks to release / ship / bump version.

If unclear, ask: "Preview only, or actually publish?"

## Pre-flight checks (always, both modes)

Run in parallel:

```bash
git rev-parse --show-toplevel       # confirm we're in the repo
git branch --show-current           # must be main
git status --porcelain              # must be empty (for publish mode)
git fetch origin main --tags        # sync tags and main
git describe --tags --abbrev=0      # find last tag (may fail if no tags)
gh auth status                      # gh CLI must be authenticated (publish mode)
```

Failure modes:

- **Not on main** — in publish mode, refuse. In preview mode, warn but proceed.
- **Dirty working tree** — in publish mode, refuse and show `git status` output. Don't auto-stash.
- **Behind origin/main** — refuse and tell the user to `git pull --ff-only`.
- **No previous tag** — that's the initial release case. Start the version at `v1.0.0` unless `package.json` says otherwise; analyze all commits.
- **`gh` not installed or not authenticated** — in publish mode, refuse with install instructions: `brew install gh && gh auth login`.

## Step 1 — Analyze commits since last tag

```bash
git log <last-tag>..HEAD --pretty=format:'%H%x09%s%x09%b%x1e' --no-merges
```

Parse each commit as conventional-commits:

- `feat(scope)?: subject` → category **Features**, signal `minor`
- `fix(scope)?: subject` → category **Bug Fixes**, signal `patch`
- `refactor(scope)?: subject` → category **Refactor**, signal `patch`
- `perf(scope)?: subject` → category **Performance**, signal `patch`
- `docs(scope)?: subject` → category **Docs**, signal `patch`
- `chore(scope)?: subject` → category **Chores**, signal `patch`
- `test(scope)?: subject` → category **Tests**, signal `patch`
- `build|ci(scope)?: subject` → category **Build/CI**, signal `patch`
- Non-conforming subject → category **Other**, signal `patch`

**Breaking change detection** (any of the following → signal `major`):
- Subject contains `!` before the colon, e.g. `feat!:` or `feat(api)!:`
- Body or footer contains a line starting with `BREAKING CHANGE:` or `BREAKING-CHANGE:`

## Step 2 — Recommend bump

Aggregate signals: `major` > `minor` > `patch`. The highest signal wins.

Compute next version from the last tag (strip leading `v`):
- `major` — `(X+1).0.0`
- `minor` — `X.(Y+1).0`
- `patch` — `X.Y.(Z+1)`

If `package.json` version disagrees with the last tag, mention it; usually the tag is the source of truth.

## Step 3 — Collect closed backlog items

Closed items can live in **two** places by the time you reach this step:

1. **Pre-archived** in `docs/backlog/_done/vX.Y.Z/*.md` — the user (or the `backlog` skill) may have already moved items into the folder matching the new version *before* running ship-release. **Check this folder first** using the recommended next version from Step 2. If it exists and has `.md` files, those are the canonical "shipped" set for this release.
2. **Still active** in `docs/backlog/*.md` with `status: done` — items finished since the last release but not yet archived.

Read both, then **union** them (dedupe by slug — pre-archived wins). The combined set is what goes into the "Closed items" section.

For each item, capture: `title`, `type`, slug (filename minus `.md`), and `branch` (for the compare link). Also remember whether it was pre-archived or still active — Step 7 only needs to move the still-active ones.

If both sources are empty, skip the "Closed items" section entirely.

> **Why check the version folder first:** the user frequently pre-archives items via the `backlog` skill (commits like `docs(backlog): reorganize done items into vX.Y.Z archive`). If you only scan top-level, you'll publish notes with an empty "Closed items" section even though the work is sitting right there in `_done/vX.Y.Z/`.

## Step 4 — Draft release notes

Backlog items are the primary source of truth for release content. Closed backlog items (from Step 3) map directly into Features / Bug Fixes / Refactor by their `type` field — they ARE the features, not a separate "Closed Items" section. Individual commits that are not already covered by a backlog item can be added to the relevant section as supplementary entries.

Generate the notes in this exact shape (English, plain markdown, no emoji unless the user asks):

```markdown
## Features
- <backlog item title, or commit subject if no covering backlog item>

## Bug Fixes
- <backlog item title, or commit subject if no covering backlog item>

## Refactor
- <backlog item title, or commit subject if no covering backlog item>

## Breaking Changes
- <description from BREAKING CHANGE footer or commit body>

**Full changelog:** https://github.com/<owner>/<repo>/compare/<last-tag>...<new-tag>
```

Rules:
- **No Highlights section. No Closed Items section.** Backlog items appear directly under Features / Bug Fixes / Refactor based on their `type` field.
- **Only include Features, Bug Fixes, Refactor, and Breaking Changes sections.** Never emit Docs, Chores, Tests, Performance, Build/CI, or Other sections — these are implementation noise, not user-facing changes.
- **No commit SHAs** in any section.
- If a commit is already represented by a backlog item, do not list it again as a separate entry.
- Omit any section with zero entries (don't print `## Features` followed by nothing).
- Use the backlog item's `title` field (not the commit subject). Capitalize the first letter.
- Always include the compare link at the end.
- Resolve `<owner>/<repo>` from `git remote get-url origin`.

## Step 5 — Present and confirm

Show the user, in one message:

1. **Last tag**: `vX.Y.Z` (date)
2. **Recommended bump**: `patch` / `minor` / `major` → **`vA.B.C`**
3. **Rationale**: one sentence on why (e.g. "1 feat commit, no breaking changes → minor bump")
4. **Drafted notes** in a fenced markdown block

Then ask the user:
- Confirm version? (or override: `patch`/`minor`/`major`/explicit `vX.Y.Z`)
- Edit notes? (offer to revise specific sections, or take their full rewrite)

Don't proceed until both are confirmed. If the user edits notes, re-present the final version once for a clean confirmation before publishing.

## Step 6 — Publish (only after explicit confirm)

Write the final notes to a temp file:

```bash
NOTES_FILE=$(mktemp -t release-notes-vX.Y.Z.XXXX.md)
# write final notes content to $NOTES_FILE
```

Bump `package.json`:

```bash
# Use the Edit tool — don't use sed or jq just to bump a version
```

Then run, sequentially:

```bash
git add package.json
git commit -m "chore(release): vX.Y.Z"
git push origin main
gh release create vX.Y.Z --target main --title "vX.Y.Z" --notes-file "$NOTES_FILE"
```

Why `gh release create` instead of `git tag && git push --tags`:
- It creates the tag remotely and attaches notes in one atomic step
- The `--notes-file` carries the AI-drafted markdown verbatim
- The release workflow triggers on the tag push regardless

`--target main` ensures the release is anchored to the just-pushed commit on main.

## Step 7 — Archive closed backlog items

Now that `vX.Y.Z` exists as a tag, move any **still-active** `docs/backlog/*.md` with `status: done` (the items from Step 3 that were *not* pre-archived) into `docs/backlog/_done/vX.Y.Z/`. Use `git mv` so the move is tracked cleanly.

```bash
mkdir -p docs/backlog/_done/vX.Y.Z
# for each still-active <slug>.md with status: done:
git mv docs/backlog/<slug>.md docs/backlog/_done/vX.Y.Z/<slug>.md
```

Then commit:

```bash
git add docs/backlog/
git commit -m "chore(backlog): archive items shipped in vX.Y.Z"
git push origin main
```

This commit goes on main *after* the tag, which is fine — the tag stays on the version-bump commit, and the archive is housekeeping that lives on the post-tag history.

**Skip this step entirely if:**
- There were zero `done` items (no commit needed), OR
- All closed items were already pre-archived in Step 3 (nothing left to move — don't make an empty commit).

## Step 8 — Report and monitor

After publishing:

1. Print the release URL — `gh release view vX.Y.Z --json url --jq .url`
2. Print the workflow run URL — `gh run list --workflow=release.yml --limit 1 --json url --jq '.[0].url'`
3. Offer to tail the run: `gh run watch` (don't run automatically — it blocks)

Then clean up the temp notes file: `rm -f "$NOTES_FILE"`.

## Initial release special case

If `git describe --tags` returns no tag:

- Default proposed version: read from `package.json`. If that's `0.1.0` or similar pre-1.0, suggest `v0.1.0` or whatever package.json says; let the user override.
- Notes: include all commits, but the "Highlights" section can be a single line like "Initial public release of agents-tracing-dashboard."
- Compare link: link to the full commit history instead of a diff, e.g. `https://github.com/<owner>/<repo>/commits/vX.Y.Z`.

## Edge cases

- **No new commits since last tag** — refuse with: "Nothing to release. Last tag vX.Y.Z is already at HEAD." Suggest preview mode if they want to see history.
- **Version in `package.json` ahead of last tag** — e.g. tag is `v1.0.7` but `package.json` says `1.1.0`. Surface the mismatch; ask whether to honor `package.json` or compute from the tag.
- **User wants to skip a version** — e.g. last tag `v1.0.7`, user asks for `v1.2.0` directly. Allow it; just confirm explicitly.
- **Pre-release tags** (`v1.0.0-rc.1`) — out of scope for the default flow. If the user asks for one, do it manually and don't try to auto-bump from a pre-release tag.

## Tone

- Lead with the recommendation, not the analysis ("Recommend `v1.0.8` (patch). Reason: 3 fix commits, no feats.")
- Show the notes in full inside a fenced block — don't summarize them, the user is going to read every line
- Keep the confirmation prompt concise: "Confirm `v1.0.8` and these notes? (or: edit notes / pick different bump / cancel)"

## Anti-patterns to avoid

- Don't push the tag first then create the release — `gh release create` does both in the right order and the workflow only needs one trigger
- Don't run `npm publish` or `pnpm pack:npm` from this skill — the workflow does it. Doubling up causes "version already exists" failures
- Don't `git tag -a vX.Y.Z` locally and `git push --tags` — that triggers the workflow without attaching AI-written notes (GH would auto-create a release with empty body)
- Don't bypass the user's confirmation, even on "looks routine" patch bumps. The whole point of this skill is human-in-the-loop notes
- Don't edit `CHANGELOG.md` — this project keeps release notes on the GitHub releases page, not in the repo
