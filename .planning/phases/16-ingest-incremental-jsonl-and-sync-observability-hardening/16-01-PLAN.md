<objective>
Create the cursor schema and safety decision layer that allows Phase 16 to prove when a Claude/Codex JSONL file can be parsed from an append offset and when it must fall back to full reparse.
</objective>

<must_haves>
- Cursor state is stored per `(source_type, file_path)` in a dedicated table.
- Cursor safety rejects truncate, rewrite, inode/device change, parser version change, missing cursor, unreadable snapshot, and `force=true`.
- Append parsing is allowed only when the file grew beyond a complete-line offset.
- The existing full parser/write path remains the fallback for every unsafe state.
- No concurrency or batching is introduced in this plan.
</must_haves>

<threat_model>
T-16-01: Stale cursor metadata could skip required parser work or corrupt session rows. Mitigation: fallback to full reparse unless all cursor safety checks pass, and update cursor only after the DB transaction succeeds.
T-16-02: Exposing local file paths in cursor/debug data can leak workstation structure. Mitigation: keep cursor data local-only, expose paths only through existing ingest debug surfaces, and never include file contents in status payloads.
</threat_model>

  <action>
Add a helper that computes the safe byte range for append parsing. It must start at `last_indexed_offset`, read forward, and return the byte offset after the last complete newline-terminated JSONL line.

Required behavior: a file ending with a partial line is not parsed past the last newline; a file with no new complete lines returns `skip_unchanged` or an empty delta decision; an append with one or more complete lines returns the exact start offset and safe end offset.

Use byte offsets, not JavaScript string length, for cursor advancement.
  </action>
  <acceptance_criteria>
    - Test proves a trailing partial line is left unparsed and cursor offset does not advance into it.
    - Test proves appending a newline-complete line advances the safe end offset to file size.
    - Test proves appending a complete line followed by a partial line advances only through the complete line.
  </acceptance_criteria>
</task>
