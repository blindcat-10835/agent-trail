# Plan 08-03: Persist message ids, tool calls, result events, and force reparse

## Goal

Fix the data-layer root cause: parser output must become SQLite rows that `assembleTurns()` can read. Also add force reparse so parser fixes can rebuild existing indexed sessions.

## Tasks

1. Update `writeSessionToDatabase()` signature to accept options while preserving existing call sites:
   - `force?: boolean`
   - `parserVersion?: string` or equivalent cache key if useful
2. Make session writes transactional.
   - Use `database.transaction()` from `better-sqlite3`.
   - Upsert `sessions`.
   - For updated sessions, delete derived rows in dependency order:
     - `tool_result_events` for this session's tool calls.
     - `tool_calls`.
     - `turns`.
     - `messages`.
   - Insert replacement messages and tool rows before commit.
3. Insert stable message ids.
   - Use `message.id` when non-empty.
   - Fallback to deterministic `${sessionId}:${ordinal}`.
   - Include `id` in the `INSERT INTO messages` column list.
   - Set `has_tool_use` for messages whose ordinal owns one or more tool calls.
4. Insert `tool_calls`.
   - Use parser metadata `messageOrdinal` for `message_ordinal`.
   - Store `tool_id`, `name`, `category`, `input_json`, `status`, `error`, `duration_ms`.
   - Capture `lastInsertRowid` for matching result event inserts.
5. Insert `tool_result_events`.
   - Insert every `TraceToolResultEvent` for the corresponding tool call.
   - Preserve timestamp and `is_partial`.
   - Decode/normalize content only as parser output already provides; never execute or inspect tool payloads as code.
6. Ensure `assembleTurns()` assigns tool calls to the expected turns.
   - If necessary, update `pairToolCalls()` to handle tool call ordinals from assistant/tool-call synthetic messages.
   - Avoid UI-side string guessing.
7. Add force reparse path through sync:
   - `syncSource(sourceType, { force: true })` or backward-compatible equivalent.
   - `writeSessionToDatabase(..., { force: true })` must bypass hash skip.
   - `POST /api/v1/sources/:type/sync` accepts `force=true` via query/body and forwards it.
8. Add tests:
   - `messages.id IS NOT NULL` after write.
   - Tool call rows exist after write.
   - Result event rows exist and link to the inserted tool call.
   - Re-sync with fewer tool calls removes stale rows.
   - Hash match skips by default but not with `force`.
   - `assembleTurns()` returns `turn.activities` containing the persisted tool call/result.

## Verification

- `pnpm test:run tests/unit/ingest/tool-persistence.test.ts tests/unit/ingest/turn-activity-regression.test.ts tests/unit/ingest/sync.test.ts`

## Acceptance Criteria

- Local DB query for synced sessions no longer returns `messages.id IS NULL`.
- `tool_calls` and `tool_result_events` are non-empty for parsed sessions with real tool usage.
- Re-sync cannot leave stale tool rows attached to a session after parser output changes.
- Force sync reparses unchanged files.
- Replay turn assembly sees structured tool activities from SQLite, not parser-only memory state.
