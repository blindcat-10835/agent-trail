```typescript
  id: string; source: TraceSource; project: string;

  id: string; ordinal: number; role: MessageRole; content: string;
  timestamp?: string; model?: string; tokenUsage?: TokenUsage;
  sourceMetadata: SourceMetadata;

  type: 'tool_call'; id: string; name: string; category: ToolCategory;
```

**Main entry point — parseCodexSession(filePath, project) → Promise<ParseResult>:**
Same structure as parseOpenClawSession(). Key differences:

**Session metadata extraction (first pass):**
- The first line of a Codex JSONL file is typically `{"type": "session_meta", ...}` with `session_id`, `cwd`, `git_branch`, `model`
- Read `session_meta.session_id` for the session UUID
- Read `session_meta.cwd` and `session_meta.git_branch` for SourceMetadata

**Response item processing (D-06, D-07):**
When `parsed.type === 'response_item'`:
- `response_item.type === 'input_text'`: create `TraceMessage` with `role: 'user'`, content from `response_item.input_text`
- `response_item.type === 'text'`: create `TraceMessage` with `role: 'assistant'`, content from `response_item.text`
- `response_item.type === 'function_call'`: create `TraceToolCall` with:
  - `id`: `response_item.call_id`
  - `name`: `response_item.name`
  - `inputJson`: `response_item.arguments` (this is a JSON string already)
  - `status: 'pending'` (result events come later from `event_msg`)
  - `category`: infer from name (reuse inferToolCategory pattern)

**Turn context tracking (D-06):**
When `parsed.type === 'turn_context'`:
- Read `parsed.turn_context.turn_id` and `parsed.turn_context.model`
- Set `currentModel = parsed.turn_context.model` — this model applies to subsequent response_items until next turn_context
- Store as `CodexTurnContext` for turn assembler reference

**Event message processing (D-07):**
When `parsed.type === 'event_msg'`:
- `event_msg.type === 'function_call_output'`: create `TraceToolResultEvent` with:
  - `content`: `event_msg.content`
  - `timestamp`: `parsed.timestamp`
  - `isPartial`: derived from event type (output vs output_final)
- Link to tool call via `event_msg.call_id` (stored for assembler pairing — D-11)

**Token count deduplication (D-09):**
Codex uses `response_item.token_count` for streaming dedup (equivalent to Claude's UUID dedup but keyed differently):
- Maintain `const messageVersions = new Map<string, { tokenCount: number; message: TraceMessage }>()`
- Key: `response_item.call_id || content_hash` for text messages, `response_item.call_id` for function_calls
- On each response_item: if existing entry has lower `token_count`, replace with new version
- On each response_item: if existing entry has equal or higher `token_count`, skip (keep existing)
- Warning: `"Line ${lineNum}: Duplicate response_item with same/lower token_count — keeping previous"`
- At the end of parsing, collect final versions from the Map

**Spawn agent handling (D-08):**
When `parsed.type === 'spawn_agent'`:
- Create `TraceSubagentLink` with:
  - `subagentSessionId`: `parsed.spawn_agent.session_id`
  - `subagentSource: 'codex'`
  - `relationship`: `parsed.spawn_agent.type === 'attached' ? 'attached' : 'spawned'`
- Add to activities array

**Session context extraction — extractCodexSessionContext(filePath, project) → SessionContext:**
- Codex JSONL filenames follow pattern `{session_id}.jsonl` in `~/.codex/sessions/`
- Read session_id from file's first JSONL line (session_meta) if available
- Fallback: extract from filename using pattern

**Error handling:**
- Catch JSON.parse errors per line → `errors.push({ line: lineNum, raw: line.substring(0, 200), error: err.message })`
- Skip lines with unknown types: `warnings.push(...)` but continue
- Never throw; always return a ParseResult

**Session construction:**
- `source: 'codex'`
- `startedAt`: first timestamp; `endedAt`: last timestamp
- `status`: endedAt exists ? 'idle' : 'active'
- `rootSessionId`/`parentSessionId`: from spawn context if this session itself was spawned
- `metrics`: populate with accumulated counts
- `cwd`/`gitBranch`: from session_meta (put in SourceMetadata)

**Token usage:**
- Codex may not provide explicit token usage; accumulate from `token_count` fields on response_items
- If no token data, `totalTokens` can be undefined

```typescript
/**
 * Parse a single Codex message line
 *
 * Helper function for testing. Only processes response_item lines;
 * session_meta, turn_context, event_msg, and spawn_agent lines return null.
 *
 * @param line - JSONL line string
 * @param context - Session context
 * @returns TraceMessage or null if not a response_item or parsing fails
 */
export function parseCodexMessage(
  line: string,
  context: SessionContext
): TraceMessage | null {
  try {
    const parsed: CodexJsonlLine = JSON.parse(line);
    if (parsed.type !== 'response_item' || !parsed.response_item) return null;
    // Only return messages for text-based response_items (not function_call — those are activities)
    if (parsed.response_item.type === 'function_call') return null;
    return parseMessage(parsed, 0, context, 0);
  } catch {
    return null;
  }
}
```

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-03-12 | Tampering | codex.ts: JSONL parsing | mitigate | try/catch per line with ParseError accumulation; malformed lines never abort full parse |
| T-03-13 | Information Disclosure | codex.ts: session_meta fields | accept | cwd/git_branch from session_meta stored in SourceMetadata; Phase 6 adds API path restrictions |
